# Архитектура

Semantic News Novelty состоит из HTTP-приложения, асинхронного ML-воркера,
хранилища, очереди и пользовательского интерфейса.

```text
Пользователь / интеграция
          │
        nginx
       ┌──┴───┐
       │      │
  Streamlit  FastAPI
                 │
        ┌────────┴────────┐
        │                 │
   PostgreSQL          RabbitMQ
   + pgvector              │
        ▲             model-service
        └─────────────────┘
```

## Компоненты

- `nginx` — единая HTTP-точка входа;
- `ui` — Streamlit-интерфейс для работы с публикациями и поиском;
- `api` — пользователи, организации, импорт, публикация, учёт операций и задачи;
- `rabbitmq` — очередь заданий обработки и семантического поиска;
- `model-service` — BGE-M3, кластеризация и классификация новизны;
- `postgres` — статьи, задания, эмбеддинги и состояние пайплайна;
- `pgvector` — cosine search по векторам размерности 1024.

## Поток публикации

1. Publisher создаёт черновик или импортирует файл.
2. Публикация переводит статьи из `draft` в `public`.
3. API создаёт задачу и отправляет в RabbitMQ только ID статей и режим.
4. `model-service` читает содержимое и историю из PostgreSQL.
5. Пайплайн рассчитывает эмбеддинги, кластеры и новизну.
6. Результат и версии моделей сохраняются в одной транзакции.
7. Обработанные публичные статьи становятся доступны поиску.

`model-service` не загружает тексты статей из внешней сети.

Для больших incremental-пачек API создаёт parent job, несколько child jobs
`vectorize` и отложенный `aggregate` job. Child jobs считают embeddings и
сохраняют их в `article_pipeline_embeddings`. Когда все child jobs завершены,
`model-service` публикует aggregate job, который загружает готовые embeddings и
обрабатывает всю исходную пачку одной модельной стадией.

## Состояния

Видимость и обработка независимы:

```text
visibility: draft | public | archived
processing: not_started | pending | processing | processed | error
```

Задача проходит состояния:

```text
queued → processing → done
                    ↘ failed
```

## Хранение ML-состояния

Для статьи сохраняются:

- BGE-M3 embedding и ревизия модели;
- `cluster_id` и происхождение назначения;
- `novelty_label` и `p_significant`;
- признаки неоднозначности и позднего поступления;
- версии пайплайна, конфигурации и классификатора.

Runtime не использует FAISS. Векторы и HNSW-индекс находятся в
PostgreSQL/pgvector.

## Полный и инкрементальный режимы

`full` совместно пересчитывает выбранный корпус и подходит для первичной
загрузки или периодического reclustering.

`incremental` сохраняет историю и последовательно добавляет новые публикации.
Он подходит для небольших поступающих батчей, но накапливает расхождение с
полным пересчётом.

Если количество ID в incremental job больше `PIPELINE_CHUNK_SIZE`, стадия
векторизации разбивается на подзадачи. Это даёт наблюдаемый прогресс по пачкам и
позволяет масштабировать embeddings-стадию несколькими воркерами, при этом
кластеризация и novelty model остаются согласованными для всей исходной пачки.

Подробности находятся в [ML-пайплайне](ml_pipeline.md).

## Границы ответственности

- API отвечает за доступ, жизненный цикл статьи и постановку задач.
- Model-service отвечает за ML-вычисления, но не за права пользователя.
- PostgreSQL является источником истины для статей и результатов.
- RabbitMQ доставляет задания, но не хранит итоговое состояние продукта.
- UI использует публичный API и не обращается к базе напрямую.
