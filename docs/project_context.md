# Контекст проекта

## 1. Название и продуктовая формулировка

Проект: **News Event Detection / Semantic News Novelty**.

Проект рассматривается как B2B2C ML-сервис для новостных редакций, платформ и медиа-сервисов.

Непосредственный клиент — редакция, новостная платформа или медиа-сервис, который может встроить систему в свой редакционный и пользовательский контур.

Конечный выгодополучатель — читатель новостей.

Сервис помогает:

- группировать похожие новости в события / сюжеты;
- находить дубли, перепечатки и близкие пересказы;
- выделять публикации, которые добавляют новую смысловую информацию;
- помогать редактору формировать более чистую новостную ленту;
- улучшать пользовательскую выдачу за счёт сгруппированных новостей, похожих материалов, блока “что нового по теме” и семантического поиска.

Проект не является попыткой создать самостоятельный публичный агрегатор новостей. Его фокус — встраиваемый ML-слой для существующей новостной платформы или редакции.

---

## 2. Цель

Цель проекта — разработать систему анализа новостного потока, которая группирует новости в события и определяет роль каждой новости внутри события:

- новое событие;
- содержательное обновление;
- дубль / перепечатка.

Бизнес-цель — снизить информационный шум и повысить качество новостной выдачи.

Для редакции это означает сокращение ручной работы по структурированию новостного потока.

Для читателя это означает более понятное представление развития темы и меньшее количество повторов.

## 4. Основные сценарии использования

### Редакторский сценарий

Редактор получает поток публикаций и должен быстро понять:

- какие новости относятся к одному событию;
- какие публикации являются дублями или перепечатками;
- какие публикации добавляют новую существенную информацию;
- какие материалы стоит вывести в ленту или связать между собой.

Сервис помогает автоматизировать эту работу.

### Пользовательский сценарий

Читатель взаимодействует не напрямую с сервисом, а с новостной платформой, в которую сервис встроен.

Для читателя результат может выглядеть как:

- сгруппированные новости по событию;
- блок “что нового по теме”;
- похожие материалы;
- семантический поиск;
- более чистая новостная лента с меньшим количеством повторов.

### Интеграционный сценарий

Новостная платформа может использовать сервис как API или внутренний компонент.

Возможные точки интеграции:

- редакторская CMS;
- поисковый сервис;
- новостная лента;
- страница сюжета;
- блок похожих материалов;
- система дайджестов;
- аналитическая панель редактора.

## 5. Данные

### Lenta.ru dataset

Путь:

```text
data/raw/lenta.csv
```

Колонки:

- `url`;
- `title`;
- `text`;
- `topic`;
- `tags`;
- `date`.

Датасет используется как основной источник новостей для проверки группировки и выделения событий.

### ru_paraphraser

Использовался для дообучения embedding-модели на парафразах.

Цель использования — улучшить поиск похожих и близких по смыслу публикаций.

---

## 6. Модель

Базовая модель:

```text
sentence-transformers/paraphrase-multilingual-mpnet-base-v2
```

Дообучение:

- metric learning на парафразах;
- фактически использован `MultipleNegativesRankingLoss`;
- `Triplet Loss` рассматривается как возможное расширение.

Модель сохранена в:

```text
models/news-flow-ru-vectorization-mpnet/final
```

Веса модели не хранятся в Git. Политика хранения описана в:

```text
docs/model_artifacts.md
```

## 7. Оценка embedding-модели

Результаты оценки retrieval-задачи:

| Модель | Recall@1 | Recall@5 | MRR@10 |
|-------|----------|----------|--------|
| Base | 0.9527 | 0.9941 | 0.9699 |
| Fine-tuned | 0.9645 | 1.0 | 0.9798 |

Вывод:

- fine-tuned модель улучшает качество поиска похожих новостей;
- retrieval-метрики подтверждают, что embedding-модель пригодна как основа для поиска близких публикаций;
- сами по себе retrieval-метрики не решают продуктовую задачу выделения ключевых обновлений, поэтому дополнительно требуется оценка роли новости внутри события.

---

## 8. Поиск

Используется FAISS CPU.

Особенности:

- embeddings нормализуются;
- similarity считается через inner product;
- nearest-neighbor поиск используется для нахождения похожих публикаций.

---

## 9. Логика событий

Одна семантическая близость недостаточна.

Похожие новости из разных лет или разных временных периодов не должны автоматически попадать в одно событие.

Событие определяется через сочетание:

- semantic similarity;
- temporal proximity.

Новая публикация присоединяется к существующему событию, если:

- у неё есть близкий сосед по embedding-пространству;
- этот сосед достаточно близок по времени;
- similarity выше заданного порога.

Если подходящего события нет, создаётся новое событие.

---

## 10. Роль новости

Для каждой новости определяется роль:

```text
new_event
update
duplicate
```

### new_event

Публикация открывает новое событие или новый сюжет.

### update

Публикация относится к уже существующему событию, но добавляет новую существенную информацию.

### duplicate

Публикация является дублем, перепечаткой или близким пересказом уже известной информации.

---

## 11. MVP

Минимальный MVP включает:

1. Подготовить Lenta dataset.
2. Очистить и нормализовать тексты.
3. Посчитать embeddings.
4. Построить FAISS index.
5. Реализовать nearest-neighbor grouping.
6. Добавить time window.
7. Присвоить роль новости: `new_event`, `update`, `duplicate`.
8. Подготовить демонстрационное представление результата.
9. Оценить качество на ручной разметке.

---

## 12. Метрики MVP

### Retrieval-метрики

- Recall@K;
- MRR;
- сравнение с baseline.

### Метрики структурирования новостного потока

- precision выделения ключевых новостей;
- recall выделения ключевых новостей;
- coverage ключевых событий;
- снижение информационного шума.

### Продуктовые метрики

- снижение времени обработки новостного потока редактором;
- снижение времени понимания темы читателем;
- число публикаций, скрытых или сгруппированных как дубли;
- число действий до понимания развития темы.

## 13. Конкурентный контекст

В конкурентном анализе важно разделять три группы.

### B2B-конкуренты

Продукты, которые продаются редакциям, медиа-командам, аналитикам или мониторинговым отделам:

- NewsWhip;
- Feedly Market Intelligence;
- Медиалогия;
- Meltwater.

Они важны для анализа функций для редакторов и аналитиков, модели распространения, цен, алертов, отчётов, workflow и ожиданий B2B-клиентов.

### UX-бенчмарки для читательской выдачи

Продукты, которые помогают понять, как конечному пользователю показываются связанные новости:

- Google News / Full Coverage;
- Яндекс/Дзен Новости;
- Ground News;
- SmartNews.

Они важны как ориентиры по группировке похожих новостей, представлению сюжета, показу разных источников, похожим материалам, пользовательскому поиску и снижению информационного шума.

### Альтернативы

Способы, которыми редакция может решать задачу без специализированного продукта:

- ручная работа редактора;
- RSS;
- Telegram-каналы;
- поисковые запросы;
- ручные таблицы;
- разовая LLM-суммаризация.

## 14. Ключевая граница проекта

Проект не должен разрастаться до самостоятельного публичного агрегатора новостей.

Граница MVP:

- не собирать весь интернет;
- не конкурировать напрямую с Google News;
- не строить полноценную consumer-платформу;
- сфокусироваться на ML-сервисе, который может быть встроен в существующую новостную платформу.

Основной фокус:

> Группировка публикаций, выявление дублей и определение смысловой новизны внутри развивающейся новостной темы.
