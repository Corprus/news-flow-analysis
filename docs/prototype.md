# Прототип системы структурирования новостей

---

## 1. Ключевая гипотеза

Если предложить редакторам новостных платформ инструмент для автоматической группировки новостей и выделения ключевых событий внутри темы для обработки новостного потока,  
то время, необходимое для структурирования новостного потока, сократится не менее чем на 30%,  
поскольку система автоматически устраняет дублирующие публикации и выделяет изменения в развитии темы.

Дополнительно предполагается, что использование системы конечными пользователями может сократить время понимания новостной темы за счёт структурированного представления информации.

## 2. Метрики

### 2.1 Основные технические метрики

### 2.1 Основные технические метрики

| Метрика | Описание | Целевое значение | Способ оценки |
|--------|----------|------------------|----------------|
| Точность выделения ключевых новостей | Доля корректно определённых ключевых новостей | ≥ 0.7 | Сравнение с ручной разметкой |
| Полнота выделения ключевых новостей | Доля найденных ключевых новостей | ≥ 0.6 | Сравнение с ручной разметкой |
| Покрытие событий | Доля ключевых событий, представленных системой | ≥ 0.8 | Анализ по размеченному набору |
| Качество поиска похожих новостей | Способность находить связанные новости | Лучше базового подхода | Сравнение с TF-IDF (Recall@K) |
---

### 2.2 Оценка достаточного объёма разметки

Для оценки точности используется вручную размеченный набор данных.

Объём выборки определяется из условия оценки точности с заданной погрешностью и доверительным уровнем 95%.

Используется формула:

n = z² * p * (1 - p) / e²

где:
- p — ожидаемое значение точности (0.7),
- z — коэффициент для доверительного уровня 95% (1.96),
- e — допустимая ошибка.

Расчёт:

- при ошибке ±10% требуется около 80 примеров,
- при ошибке ±8% — около 120–130,
- при ошибке ±5% — более 300.

В рамках проекта используется следующий подход:

- пилотная разметка: 30–50 новостей,
- расширенная разметка: 150–250 новостей.

Это позволяет получить устойчивую оценку точности с погрешностью порядка ±8–10%.

---

### 2.3 Разметка данных

Разметка выполняется вручную по следующим правилам:

- первая новость по теме считается ключевой,
- новость считается ключевой, если содержит новую информацию,
- дублирующие и перефразированные публикации не считаются ключевыми.

Цель разметки — оценка и настройка алгоритма.

---

### 2.4 Продуктовые метрики (проектируемые)

#### Время обработки новостного потока редактором

Включает:
- группировку новостей,
- выделение ключевых событий.

Ожидаемое снижение: не менее 30%.

Оценка проводится через сравнение времени выполнения задачи по обработке новостного потока:

1. вручную,
2. с использованием системы.

Измеряется время, необходимое для группировки новостей и выделения ключевых событий.

Метрика рассчитывается как отношение времени выполнения задачи с системой к времени без неё.

Ожидается снижение времени не менее чем на 30%.

---

#### Время понимания новостной темы пользователем

Описывает время, необходимое для понимания ключевых событий.

Рассматривается как дополнительная метрика без строгой количественной оценки.

Пользователю предлагается набор новостей по одной теме.

Измеряется время, необходимое для понимания ключевых событий.

Для проверки корректности понимания задаются уточняющие вопросы.

Сравниваются два сценария:
- обычный список новостей,
- структурированное представление.

Метрика рассчитывается как среднее время успешного выполнения задачи.

## 3. Scope прототипа

### Входит в прототип

| Компонент | Описание |
|----------|----------|
| Данные | Используется датасет новостей Lenta |
| Обработка текста | Расчёт семантических представлений |
| Кластеризация | Группировка новостей по темам |
| Выделение ключевых событий | Определение новых событий внутри темы |
| Оценка качества | Проверка на размеченном наборе |
| Интерфейс | Демонстрация результатов (Streamlit / API) |

---

### Не входит в прототип

| Ограничение | Причина |
|------------|--------|
| Сбор новостей из источников | Не влияет на проверку гипотезы |
| Масштабируемая инфраструктура | Прототип |
| Персонализация | Вне scope |
| Обучение модели с нуля | Используются готовые модели |
| Production-интерфейс | Демонстрационный режим |

## 6. Формат прототипа

На вход подаётся набор новостей.

На выходе формируются:
- группы новостей по темам,
- ключевые события внутри каждой темы,
- связанные публикации.

## 8. Критерии остановки

Прототип считается завершённым, если:

- достигнуты целевые значения точности и полноты,
- обеспечено покрытие ключевых событий,
- результаты устойчивы на нескольких темах.

Прототип также может быть остановлен, если достичь целевых метрик не удаётся.
