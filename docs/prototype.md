# Прототип системы структурирования новостей

---

## 1. Ключевая гипотеза

Если предложить редакторам новостных платформ инструмент для автоматической группировки новостей, выявления дублей и выделения ключевых обновлений внутри темы, то время, необходимое для структурирования новостного потока, сократится не менее чем на 30%.

Это возможно, поскольку система автоматически:

- группирует связанные публикации в события;
- устраняет часть дублирующих и близких по смыслу материалов;
- выделяет публикации, которые содержат новую существенную информацию;
- помогает редактору быстрее понять развитие темы.

Дополнительно предполагается, что использование результатов системы в пользовательской новостной выдаче может сократить время понимания темы конечным читателем.

Для читателя продукт может проявляться как:

- сгруппированные новости по событию;
- блок “что нового по теме”;
- похожие материалы;
- семантический поиск;
- уменьшение количества повторов в ленте.

Проект не предполагает создание самостоятельного публичного агрегатора новостей. Прототип рассматривается как ML-сервис, который может быть встроен в редакционный и пользовательский контур существующей новостной платформы.

## 2. Метрики

### 2.1 Основные технические метрики

| Метрика | Описание | Целевое значение | Способ оценки |
|--------|----------|------------------|---------------|
| Точность выделения ключевых новостей | Доля корректно определённых ключевых новостей среди всех новостей, которые система пометила как ключевые | ≥ 0.7 | Сравнение с ручной разметкой |
| Полнота выделения ключевых новостей | Доля найденных системой ключевых новостей среди всех ключевых новостей в ручной разметке | ≥ 0.6 | Сравнение с ручной разметкой |
| Покрытие событий | Доля ключевых событий, представленных системой | ≥ 0.8 | Анализ по размеченному набору |
| Семантический поиск | Качество ранжирования результатов поиска по смыслу | `MRR@10` выше уровня бейзлана | Отдельная поисковая тестовая выборка с оценкой релевантности |
| Семантический поиск | Качество ранжирования результатов поиска по смыслу | `MRR@10` выше уровня бейзлана | Отдельная поисковая тестовая выборка с оценкой релевантности |

Текущий технический смоук-тест в `notebooks/09_search_and_dedup_benchmark.ipynb` получил метрики:

- базовый поиск TF-IDF по ключевым словам: `MRR@10 = 0.9067`;
- BGE-M3: `MRR@10 = 0.9926`;
- BGE-M3 с collapse по предсказанным `exp_10a` кластерам: `MRR@10 = 0.9944`;
- снижение доли повторов в Top-10 относительно TF-IDF: `75.14%`.

Результат рассчитан на 119 запросах, синтетически сформированных из заголовков
golden-публикаций. Он подтверждает техническую реализуемость, но не заменяет продуктовую
проверку на реальных запросах аналитиков.

### 2.2 Оценка достаточного объёма разметки

Для оценки точности используется вручную размеченный набор данных.

Объём выборки определяется из условия оценки точности с заданной погрешностью и доверительным уровнем 95%.

Используется формула:

```text
n = z² * p * (1 - p) / e²
```

где:

- `p` — ожидаемое значение точности;
- `z` — коэффициент для доверительного уровня 95%;
- `e` — допустимая ошибка.

Расчёт при ожидаемой точности около 0.7:

- при ошибке ±10% требуется около 80 примеров;
- при ошибке ±8% требуется около 120–130 примеров;
- при ошибке ±5% требуется более 300 примеров.

В рамках проекта используется следующий подход:

- пилотная разметка: 30–50 новостей;
- расширенная разметка: 150–250 новостей.

Это позволяет получить первичную оценку качества с погрешностью порядка ±8–10%.

---

### 2.3 Разметка данных

Разметка выполняется вручную по следующим правилам:

- первая новость по событию считается ключевой;
- новость считается ключевой, если содержит новую существенную информацию;
- дублирующие публикации, перепечатки и близкие перефразирования не считаются ключевыми;
- похожие публикации из разных временных периодов не должны автоматически объединяться в одно событие;
- событие определяется не только семантической близостью, но и временной близостью.

Цель разметки — оценка и настройка алгоритма группировки и определения роли новости.

### 2.4 Продуктовые метрики

#### Время обработки новостного потока редактором

Метрика показывает, насколько система снижает трудозатраты редактора.

В задачу редактора входит:

- просмотр набора публикаций;
- группировка связанных новостей;
- выделение дублей;
- определение публикаций, которые содержат новые существенные факты.

Оценка проводится через сравнение двух сценариев:

1. ручная обработка новостного потока;
2. обработка того же потока с использованием системы.

Целевое значение: снижение времени обработки не менее чем на 30%.

#### Время понимания новостной темы пользователем

Метрика показывает, помогает ли структурированное представление быстрее понять развитие темы.

Сравниваются два сценария:

1. обычный список новостей;
2. структурированное представление: событие, ключевые публикации, дубли, обновления и краткое указание, что изменилось.

Метрика рассматривается как дополнительная, так как в рамках MVP основной фокус — проверка редакторского сценария.

#### Снижение информационного шума

Метрика показывает, насколько система уменьшает количество публикаций, которые пользователю или редактору нужно просматривать вручную.

Возможные способы оценки:

- доля публикаций, сгруппированных как дубли;
- коэффициент сокращения (reduction rate);
- число публикаций в исходном потоке по сравнению с числом ключевых публикаций после обработки.

Важно: коэффициент сокращения не является самостоятельной целевой метрикой. Высокое сжатие полезно только при сохранении охвата (coverage) ключевых событий.

## 3. Границы прототипа

### Входит в прототип

| Компонент | Описание |
|----------|----------|
| Данные | Используется датасет новостей Lenta |
| Обработка текста | Очистка и подготовка текстов |
| Векторизация | Расчёт семантических представлений новостей |
| Поиск похожих публикаций | Использование FAISS / nearest-neighbor поиска |
| Группировка | Объединение новостей в события с учётом семантической и временной близости |
| Определение роли новости | Классификация роли публикации: `new_event`, `update`, `duplicate` |
| Оценка качества | Проверка на размеченном наборе |
| Интерфейс | Демонстрация результатов через Streamlit / API / notebook |

### Не входит в прототип

| Ограничение | Причина |
|------------|---------|
| Сбор новостей из внешних источников | Не влияет на проверку основной гипотезы |
| Масштабируемая production-инфраструктура | Прототип проверяет алгоритмическую и продуктовую гипотезу |
| Полная персонализация | Вне границ MVP |
| Обучение модели с нуля | Используются готовые модели и дообучение |
| Production-интерфейс для редакции | На этапе прототипа достаточно демонстрационного режима |
| Самостоятельный публичный агрегатор | Продукт рассматривается как B2B2C-сервис для встраивания в существующую платформу |
| Полная LLM-суммаризация новостей | Возможное расширение, но не ядро текущего прототипа |

### Готовые решения

| Компонент | Назначение |
|----------|-----------|

### Собственная реализация

| Компонент | Назначение |
|----------|-----------|
| Pipeline обработки новостей | Объединение всех этапов |
| Алгоритм группировки | Объединение публикаций в события |
| Учет временного окна | Разделение похожих, но относящихся к разным периодам публикаций |
| Алгоритм выделения ключевых событий | Определение новой информации внутри темы |
| Присвоение роли новости | Разметка публикаций как `new_event`, `update`, `duplicate` |
| Калибровка параметров | Настройка порогов similarity и временного окна |

### Архитектура

Используется ранее разработанный конвейер обработки данных из проекта семантического поиска с адаптацией под задачу новостей.

Базовый pipeline:

1. Загрузка новостей.
2. Очистка и нормализация.
3. Расчёт эмбеддингов.
4. Построение FAISS index.
5. Поиск ближайших соседей.
6. Группировка с учётом времени.
7. Определение роли новости.
8. Вывод результатов для анализа.

## 5. Риски

| Тип риска | Описание | Возможное решение |
|----------|----------|-------------------|
| Модель | Семантическая близость не всегда отражает новизну | Разделить поиск кандидатов и определение новизны; использовать ручную разметку |
| Данные / доменный перенос | Модель может показывать приемлемое качество на фиксированном учебном датасете, но хуже переноситься на реальные новостные данные из-за отличий в стиле, источниках, тематиках и структуре дубликатов | Размечать несколько тем и временных периодов |
| Кластеры | Новости могут группироваться слишком широко | Использовать временное окно и дополнительные эвристики |
| Продукт | Редакторы не получают ожидаемого выигрыша по времени | Проверять продуктовую метрику времени обработки |
| UX | Конечному пользователю может быть непонятно, почему публикация считается новой | Добавить объяснение: что именно изменилось |
| Границы проекта | Проект может разрастись до полноценного агрегатора | Ограничить MVP задачей встраиваемого ML-сервиса |

### Проверка качества выборки и доменного переноса

Чтобы снизить риск нерепрезентативной оценки, тестовая разметка формируется не из одной темы, а из нескольких независимых тематических групп новостей.

Минимальные требования к проверочной выборке:

- не менее 5–7 разных тематик;
- в каждой тематике должны быть как дублирующие / перефразированные публикации, так и новости с новой информацией;
- в выборке должны присутствовать публикации разных дат и разных источников;
- часть данных должна быть отложена и не использоваться при настройке порогов и параметров алгоритма.

Дополнительно качество проверяется отдельно:

1. на данных, близких к обучающему / тестовому датасету;
2. на более реалистичной выборке новостей, имитирующей входящий поток клиента.

Если качество на фиксированном датасете достигается, но заметно падает на реалистичной выборке, риск доменного переноса считается подтверждённым.

## 6. Формат прототипа

На вход подаётся набор новостей.

На выходе формируются:

- группы новостей по событиям;
- ключевые публикации внутри каждой группы;
- дубли и перепечатки;
- обновления, содержащие новую информацию;
- связанные публикации;
- демонстрационное представление результата для редактора или читателя.

Пример результата:

```text
Событие: ...
- первая ключевая публикация
- обновление: добавлена новая информация
- дубли / перепечатки
- связанные материалы
```

## 7. Бизнес-ценность

### Для редакции

- снижение времени обработки новостного потока;
- уменьшение объёма ручной работы;
- автоматическое выявление дублей и перепечаток;
- помощь в выборе публикаций для ленты;
- более прозрачное понимание развития темы.

### Для платформы

- возможность обрабатывать больше новостей тем же составом команды;
- повышение качества новостной выдачи;
- улучшение поиска и навигации по связанным материалам;
- повышение полезности продукта для читателя;
- потенциальное увеличение вовлечённости за счёт более чистой структуры новостей.

### Для читателя

- меньше повторов в новостной ленте;
- быстрее видно, что действительно изменилось;
- проще понять развитие события;
- удобнее находить похожие материалы;
- появляется возможность семантического поиска по новостному архиву.

## 8. Критерии остановки

Прототип считается успешно завершённым, если в течение ограниченного периода разработки и тестирования выполнены следующие условия:

- достигнуты целевые значения основных технических метрик:
  - точность выделения ключевых новостей (`precision`) ≥ 0.7;
  - полнота выделения ключевых новостей (`recall`) ≥ 0.6;
  - покрытие ключевых событий (`coverage`) ≥ 0.8;
- качество семантического поиска выше уровня качества бейзлайна по MRR@10;
- результаты устойчивы минимум на нескольких разных тематических группах новостей;
- на отложенной проверочной выборке не наблюдается существенного падения качества по сравнению с выборкой, использованной для настройки параметров.

### Текущий статус

Минимальные технические критерии MVP достигнуты:

| Метрика | Требование | Получено |
|---|---:|---:|
| Попарный F1 кластеризации | ≥ 0.75–0.80 | **0.9017** |
| Доля ложных объединений | ≤ 10–15% | **1.89%** |
| F1 значимых обновлений | ≥ 0.75 | **0.9114** |
| Recall значимых обновлений | ≥ 0.80 | **0.9863** |
| MRR@10 семантического поиска | выше уровня бейзлайна | **0.9944 против 0.9067** |
| Снижение повторов в Top-10 | ≥ 30% | **75.14%** |

Результат закрывает технический критерий остановки на текущем golden-наборе. Он не
закрывает риск доменного переноса: перед продуктовым выводом требуется пилот на данных
клиента и реальных поисковых запросах.

### Критерий неуспешного завершения

Тест гипотезы считается неуспешным, если после 2–3 итераций настройки алгоритма и разметки расширенной выборки не удаётся достичь минимально допустимых метрик:

- точность < 0.65;
- полнота < 0.55;
- покрытие событий < 0.7;
- семантический поиск не превосходит уровень бейзлайна по MRR@10;
- качество заметно падает при проверке на реалистичной выборке новостей, отличающейся от исходного датасета.

Максимальный срок проверки прототипа — 3–4 недели активной разработки и тестирования. Если за этот срок после нескольких итераций улучшения модели и настройки порогов целевые или минимально допустимые метрики не достигаются, гипотеза о пригодности выбранного подхода для задачи считается неподтверждённой.

## 9. Модель монетизации

| Модель | Описание |
|-------|----------|
| B2B SaaS | Подписка для новостной платформы или редакции |
| API | Доступ к функциональности группировки, поиска похожих публикаций и определения новизны |
| Лицензия | Использование сервиса в инфраструктуре клиента |
| Кастомизация | Настройка и интеграция под задачи конкретной редакции |
| Private cloud / on-premise | Возможный enterprise-вариант для клиентов с требованиями к приватности и инфраструктуре |

Основная гипотеза монетизации — B2B SaaS с оплатой по объёму данных, числу источников, глубине истории или количеству API-запросов.

On-premise deployment рассматривается не как базовое требование MVP, а как потенциальное конкурентное преимущество для enterprise-клиентов.
