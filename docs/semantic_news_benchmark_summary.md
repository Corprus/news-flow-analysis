# Выводы по итогам конкурентного анализа

Проект: **News Event Detection / Semantic News Novelty**

## 1. Контекст продукта

Проект рассматривается как **B2B2C ML-сервис для новостных редакций и платформ**.

Непосредственный клиент — редакция, новостная платформа или медиа-сервис. Сервис встраивается в редакционный и пользовательский контур существующего продукта: редакторскую админку, новостную ленту, поиск, блок похожих материалов или страницу сюжета.

Конечный выгодополучатель — читатель новостей. Для него продукт должен проявляться как более чистая новостная выдача: меньше повторов, сгруппированные новости по событию, блок «что нового по теме», похожие материалы и семантический поиск.

Проект **не является самостоятельным публичным агрегатором новостей**. Его цель — не конкурировать напрямую с Google News или Яндекс/Дзен Новостями, а предоставить редакции ML-инструмент для структурирования собственного новостного потока.

В конкурентном анализе были рассмотрены три основных B2B-ориентира:

| Конкурент | Роль в анализе | Что показывает для нашего продукта |
|---|---|---|
| NewsWhip Spike | Прямой / наиболее близкий B2B-конкурент | Стандарт по скорости обработки, мониторингу в реальном времени, алертам, таймлайну развития истории и API |
| Feedly Market Intelligence | Частично прямой конкурент | Стандарт по AI-фильтрации, дайджестам, синтезу информации, дашбордам и командной работе |
| Медиалогия | Косвенный B2B-конкурент / локальный ориентир | Стандарт по локальному мониторингу СМИ и соцсетей, уведомлениям, отчётам, источникам и работе с российским рынком |

Главный вывод анализа: конкуренты задают ожидания по скорости, мониторингу, уведомлениям, отчётам и интеграциям, но их основной фокус отличается от нашего. NewsWhip делает упор на вовлечённость и прогнозирование интереса аудитории, Feedly — на широкий market intelligence, Медиалогия — на PR/SMM и мониторинг инфополя. Ниша Semantic News Novelty — **структурирование новостного потока для редакции и выявление смысловой новизны внутри события**.

## 2. Минимальные требования к функциональности

Минимально конкурентоспособный продукт должен решать не только техническую задачу поиска похожих текстов, но и прикладную задачу редакции: быстро понять, какие публикации относятся к одному событию, где дубли, а где действительно появилось новое содержание.

| Требование | Описание | Приоритет |
|---|---|---|
| Группировка публикаций в события / сюжеты | Система должна объединять связанные новости в осмысленные событийные кластеры, а не просто показывать сырой поток публикаций. | Must-have |
| Поиск похожих публикаций | Необходима векторная модель и поиск похожих новостей, чтобы находить дубли, пересказы и связанные материалы. | Must-have |
| Выявление дублей и перепечаток | Система должна определять публикации, которые не добавляют новой информации и являются повтором уже известного материала. | Must-have |
| Определение содержательных обновлений | Система должна отличать публикации, которые действительно добавляют новую существенную информацию внутри события. | Must-have |
| Присвоение роли новости | Каждая новость должна получать роль: `new_event`, `update` или `duplicate`. | Must-have |
| Учет временного контекста | Похожие новости из разных временных периодов не должны автоматически объединяться в одно событие. Нужна связка семантической близости и временной близости. | Must-have |
| API или редакторский интерфейс | Так как клиентом является редакция/платформа, результат должен быть доступен через API, демонстрационный интерфейс или интеграцию с редакционной системой. | Must-have |
| Базовое представление для конечного читателя | Результат должен быть пригоден для пользовательской выдачи: сгруппированные материалы, похожие новости, блок «что нового по теме». | Should-have |
| Уведомления и дайджесты | Для редакции полезны уведомления о новых событиях и важных обновлениях, а также регулярные сводки. | Should-have |
| Семантический поиск | Пользователь или редактор может искать материалы по смыслу, а не только по ключевым словам. | Should-have |
| Private cloud / on-premise | Возможный вариант для крупных клиентов с требованиями к безопасности и инфраструктуре. Для MVP не является обязательным. | Could-have |

Итоговый стандарт по функциональности: MVP должен уметь строить события, находить похожие публикации, отделять дубли от обновлений, учитывать время и предоставлять результат в виде, пригодном для редакторского сценария.

## 3. Ключевые конкурентные преимущества

Ключевое отличие продукта — фокус не на мониторинге как таковом, а на **смысловой новизне** внутри развивающейся новостной темы.

| Конкурентное преимущество | Суть преимущества | Почему это важно |
|---|---|---|
| Оценка смысловой новизны | Продукт определяет, добавляет ли публикация новую информацию по сравнению с уже известными материалами в событии. | Это отличает продукт от систем, которые приоритизируют новости по популярности, упоминаниям или вовлечённости. |
| Объяснение новизны | Система должна показывать, почему публикация считается обновлением или дублем: что именно изменилось, какой новый факт появился. | Редактору нужно доверять модели и понимать её решение. |
| Узкий редакционный фокус | Продукт не пытается быть широким PR/SMM или market intelligence-комбайном, а решает конкретную задачу новостной ленты. | Узкий фокус делает продукт проще, понятнее и потенциально дешевле для редакций. |
| Снижение информационного шума | Система группирует повторы и показывает ключевые обновления, уменьшая количество материалов, которые нужно просматривать вручную. | Это основная ценность и для редакции, и для конечного читателя. |
| B2B2C-эффект | Продукт помогает редакции, но результат видит читатель: сгруппированные сюжеты, похожие материалы, «что нового по теме», семантический поиск. | Это связывает внутреннюю эффективность редакции с качеством пользовательского опыта. |
| Встраиваемость | Продукт может работать как API или модуль внутри существующей новостной платформы. | Это снимает необходимость строить собственный агрегатор и позволяет продавать технологию существующим медиа. |
| Потенциально более доступный порог входа | Продукт решает более узкую задачу, чем широкие media intelligence-платформы. | Это даёт шанс быть доступнее для небольших и средних редакций. |

Главное позиционирование:

> Semantic News Novelty — это не система «найти самые популярные новости», а система «понять, что действительно изменилось в развитии события».

## 4. Оптимальная модель ценообразования и монетизации

Наиболее подходящая модель — **B2B SaaS / API-подписка**.

Такой вариант соответствует рынку: рассмотренные конкуренты также распространяются как B2B SaaS, подписка или sales-led enterprise-продукт. При этом Semantic News Novelty должен позиционироваться не как широкий media intelligence-комбайн, а как более узкий и специализированный сервис для редакций и новостных платформ.

### 4.1 Базовая модель

| Элемент модели | Рекомендация |
|---|---|
| Основная модель | B2B SaaS / API subscription |
| Клиент | Редакция, новостная платформа, медиа-сервис |
| Единица ценности | Обработанные публикации, источники, тематические ленты, глубина истории, API-запросы |
| Базовый способ продажи | Пилот / демо на данных клиента, затем подписка |
| Расширенный вариант | Enterprise-пакет с SLA, CMS-интеграцией, private cloud или on-premise |

### 4.2 Возможные параметры тарификации

Стоимость может зависеть от:

- объёма обработанных публикаций;
- количества подключённых источников;
- количества тематических лент;
- глубины истории;
- количества API-запросов;
- числа пользователей редакторского интерфейса;
- требований к SLA;
- необходимости интеграции с CMS;
- варианта развёртывания: SaaS, private cloud, on-premise.

### 4.3 Рекомендуемая структура тарифов

| Тариф | Для кого | Логика ограничения |
|---|---|---|
| Demo / Pilot | Проверка ценности на данных клиента | Ограничение по объёму данных, источникам и сроку использования |
| Basic / Pro | Небольшие редакции или тематические медиа | Базовые кластеры, поиск похожих публикаций, ограниченная история, базовый API |
| Team / Business | Редакции и новостные платформы | Больше источников, история, уведомления, редакторский интерфейс, отчёты |
| Enterprise | Крупные клиенты | SLA, интеграция с CMS, кастомные источники, private cloud / on-premise |

### 4.4 Ценовой ориентир

По конкурентам виден широкий разброс стоимости:

| Конкурент | Оценка минимальной месячной стоимости |
|---|---:|
| Медиалогия | около $300/month |
| Feedly Market Intelligence | около $1,200/month |
| NewsWhip Spike | около $2,000/month или $2,000/user/month |

Эти цены нельзя сравнивать напрямую: у конкурентов различаются единицы тарификации, лимиты, число пользователей, глубина истории, SLA и состав функций.

Рабочий вывод для Semantic News Novelty:

> Базовый тариф должен быть ниже широких enterprise media intelligence-платформ, потому что продукт решает более узкую задачу. Рабочий ориентир для стартового пакета — ниже уровня $1,200–2,000/month, потенциально в диапазоне $300–800/month для небольших редакций. Для крупных клиентов возможна отдельная enterprise-цена.

## 5. Минимальные показатели и бенчмарки

Бенчмарки разделены на четыре группы:

1. ML-качество.
2. Продуктовая польза для редакции.
3. Пользовательский опыт.
4. Интеграционные и операционные показатели.

| Категория | Метрика / стандарт | Минимальный ориентир | Как проверять |
|---|---|---:|---|
| Retrieval | Recall@1 для поиска дублей / парафразов | > 0.9 | Тест на парафразах и похожих новостях |
| Retrieval | Recall@K / MRR | Выше baseline | Сравнение с TF-IDF и базовой embedding-моделью |
| ML / продуктовая классификация | Precision выделения ключевых новостей | ≥ 0.7 | Сравнение с ручной разметкой |
| ML / продуктовая классификация | Recall выделения ключевых новостей | ≥ 0.6 | Сравнение с ручной разметкой |
| Coverage | Покрытие ключевых событий | ≥ 0.8 | Доля ключевых событий, представленных системой |
| Дедупликация | Корректное выявление дублей и перепечаток | Отдельно оценивать precision/recall по дублям | Ручная разметка duplicate / non-duplicate |
| Скорость | Время обработки новой публикации | 1–5 минут для MVP | Измерение времени от попадания новости в систему до присвоения кластера и роли |
| Продуктовая польза | Снижение времени обработки новостного потока редактором | ≥ 30% | Сравнение ручного сценария и сценария с системой |
| Информационный шум | Reduction ratio | Только вместе с coverage | Сравнивать сжатие потока и сохранение ключевых событий |
| UX | Количество действий до понимания структуры события | 1–2 действия | UX-тестирование редакторского или пользовательского сценария |
| Объяснимость | Доля понятных объяснений для update/duplicate | Целевой ориентир для MVP можно задать после пилотной разметки | Ручная оценка объяснений редактором |
| Интеграция | API или базовый сервисный интерфейс | Должен быть в MVP | Проверка интеграционного сценария |

Важно: **reduction ratio не должен использоваться как самостоятельная целевая метрика**. Высокое сжатие потока полезно только в том случае, если сохраняется coverage ключевых событий. Иначе система может просто скрыть важные новости вместе с дублями.

## 6. Итоговые требования к продукту

На основе анализа конкурентов и выделенных бенчмарков минимально конкурентоспособный Semantic News Novelty должен обладать следующими характеристиками:

| Блок | Итоговое требование |
|---|---|
| Функциональность | Группировать публикации в события, находить похожие новости, отделять дубли от обновлений, учитывать время, присваивать роли `new_event`, `update`, `duplicate`. |
| Ключевое отличие | Делать акцент на смысловой новизне, а не только на популярности, упоминаниях или вовлечённости. |
| Пользовательская ценность | Снижать информационный шум для редактора и конечного читателя. |
| Редакторский сценарий | Показывать структуру события: ключевая публикация → обновления → дубли. |
| Читательский сценарий | Давать основу для блоков «что нового по теме», похожих материалов, семантического поиска и сгруппированной ленты. |
| Метрики качества | Precision ≥ 0.7, recall ≥ 0.6, coverage ≥ 0.8, Recall@1 > 0.9, снижение времени редактора ≥ 30%. |
| Монетизация | B2B SaaS / API-подписка с оплатой по объёму, источникам, API-запросам или глубине истории. |
| Интеграция | API / сервисный интерфейс как базовое требование; CMS-интеграция и private deployment как развитие. |

Финальный вывод:

> Бенчмаркинг показывает, что для конкурентоспособности продукт должен соответствовать рыночным ожиданиям по скорости, интеграции, уведомлениям и отчётности, но отличаться более узким фокусом на редакционную задачу: группировать новости в события, отделять дубли от обновлений и выявлять новую смысловую информацию. Это позволяет сформулировать конкретные требования к MVP и метрики, которые нужно считать дальше.
